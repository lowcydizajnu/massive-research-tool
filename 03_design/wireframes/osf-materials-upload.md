# Wireframe spec — OSF materials upload (Study Record panel)

- **Serves user flow:** [Finish a study and publish its record](../../02_product/user-flows/finish-a-study-and-publish-its-record.md)
- **IA placement:** [Information architecture](../ia/information-architecture.md)
- **Persona:** [Hanna Kowalczyk — postdoc operator](../../02_product/personas/postdoc-operator.md)
- **Status:** draft

## Purpose

Let a researcher push their study's **materials** — the stimulus files it uses, the machine-readable design snapshot, and a human-readable protocol PDF — to their **OSF project**, in one click, so their preregistration/record is reproducible. (Implements ADR-0094. Copies "Replicate/Preregistration" researcher vocabulary, not "node/WaterButler".)

## Layout

A card in the Study Record's OSF section, directly beneath the existing **Push to OSF** (record summary) affordance:

- **Header row** — title "Materials on OSF" + a one-line explainer ("Send your stimuli, design, and a protocol PDF to your OSF project.").
- **Artifact list** — one row per uploadable item: an icon + name + a right-aligned **status pill**. Groups: *Stimuli* (0-N files from the study), then *Design snapshot* (`design-snapshot.json`), then *Protocol* (`protocol.pdf`).
- **Footer row** — the primary **Upload to OSF** button (or **Re-upload** once pushed) on the left; a privacy line on the right ("Files go to your OSF project — private unless you make it public. Participant responses are never included.").

## Content inventory

- **Card title** — "Materials on OSF" — static.
- **Explainer** — static one-liner.
- **Artifact row × N** — name (computed: humanized block label + filename for stimuli; fixed names for JSON/PDF), kind icon (image/audio/video/document/json/pdf), size (computed, from `workspace_material` / generated bytes), **status pill** (computed: *Not uploaded* / *Uploading…* / *Uploaded* + link to the file on OSF / *Failed* + reason / *Skipped — too large*). Max name length ~2 lines, truncate with title tooltip.
- **Upload / Re-upload button** — primary; label depends on prior push state.
- **Privacy line** — static.
- **No-connection notice** — when OSF isn't connected: "Connect OSF in Settings · Connections to upload materials." with a link.
- **No-node notice** — when the study has never been pushed to OSF: "Create your OSF record or preregister first — materials attach to your OSF project." with a link to the record/preregister action.
- **Last-uploaded timestamp** — computed, shown after a successful push ("Last uploaded 2 minutes ago").

## States

- **Default (connected, node exists, not yet uploaded)** — artifact list with every row *Not uploaded*; button reads **Upload to OSF**, enabled.
- **Loading** — while the record/materials query resolves: skeleton rows.
- **Empty** — study has **no** stimulus files: still show the *Design snapshot* + *Protocol* rows (those always exist); a muted "This study has no uploaded stimuli — the design and protocol will still be sent." Button enabled.
- **No connection** — list is muted/disabled; the no-connection notice replaces the button.
- **No OSF node** — list is muted/disabled; the no-node notice replaces the button.
- **Uploading (partial)** — button shows pending ("Uploading…", spinner, disabled); each row flips *Not uploaded → Uploading… → Uploaded/Failed* as the server reports; no per-file cancel in v1.
- **Success** — rows show *Uploaded* with a file link; button becomes **Re-upload**; last-uploaded timestamp appears.
- **Partial failure** — some rows *Uploaded*, some *Failed* (with reason) or *Skipped — too large*; a summary line "3 of 5 uploaded"; button stays **Re-upload** (re-running retries only the not-yet-uploaded/failed ones, updating versions for changed files).
- **Error (whole push)** — e.g. OSF rejected the token (401): an inline error "OSF rejected your connection — reconnect in Settings · Connections." Rows revert to prior state.

## Interactions

- **Upload to OSF / Re-upload** — click → calls the `uploadMaterials` mutation → button pending; on resolve, per-file statuses + timestamp update; toast "Materials uploaded to OSF" (or "3 of 5 uploaded — see details"). Error path: inline message + toast, rows unchanged.
- **File link (Uploaded row)** — opens the file on osf.io in a new tab (`rel="noopener"`).
- **No-connection / no-node link** — navigates to Settings · Connections / the record-create affordance.

## Edge cases

- **Very long filenames / block labels** — truncate to 2 lines + `title` tooltip.
- **Zero stimuli** — the empty state above (JSON + PDF still upload).
- **Many stimuli (10-50+)** — list scrolls inside the card (`overflow-y:auto`, capped height); the mutation processes sequentially, so the upload can take a while — the pending button + per-row progress carry the feedback.
- **Oversized file** — a stimulus above the per-file cap is marked *Skipped — too large* (never streamed through the function) rather than failing the whole push.
- **Slow network / long upload** — the inline mutation may run for tens of seconds; keep the button in its pending state; do not let a stage-nav click discard progress silently.
- **Name collision on OSF** — handled server-side (update existing file → new version); the row still resolves to *Uploaded*, not a duplicate.
- **Permissions denied** — only owner/admin/editor (write roles) see an enabled button; viewers see the list read-only with the privacy line, no button.
- **Offline** — the mutation fails fast with the standard offline toast; rows unchanged.

## Accessibility notes

- Status pills carry text (not colour-only); *Failed*/*Skipped* also state the reason in the row so a screen reader conveys it.
- The upload button uses the shared `PendingButton` (announces busy state); focus stays on the button through the pending→resolved transition.
- Artifact rows are a list (`<ul>`/`<li>`); file links have discernible names ("Open <filename> on OSF").
- Live region announces the summary result ("3 of 5 materials uploaded") on completion.

## Open questions

- Should re-upload diff bytes and skip unchanged files, or always PUT a new version? (v1: PUT known files as new versions; a content hash to skip unchanged is a later optimization.)
- Do we want a per-file "exclude from OSF" toggle (e.g. omit a specific stimulus)? Deferred — v1 uploads all stimuli.
- Auto-upload at preregistration (ADR-0094 Option C) — surfaced here later as an opt-in checkbox if demand appears.

# JTBD — Get set up

> **Status:** draft (2026-05-28). Covers the "I'm new here, I need to be ready to do real work" job. Distinct from `build-a-study` because the work shape is different — onboarding decisions stick for the life of the workspace.

## The job, in one sentence

**When** I (researcher, postdoc, or lab admin) first arrive on this product or come back to manage my workspace,
**I want to** establish identity, theme preference, workspace membership, and the connections my future studies will rely on (OSF, providers, payment) without losing momentum,
**so I can** focus on the actual research instead of plumbing — and so the people in my workspace see a coherent space rather than a sequence of half-configured screens.

## Forces

**Pushing toward:** a real project waiting (preregistration deadline, paper draft); existing OSF account; a PI's directive to use this tool; curiosity about Frameworks; the user just got an invite link.

**Pulling away:** SaaS onboarding fatigue; suspicion of yet-another-account; resistance to connecting third-party services; OAuth scope anxiety; "I'll configure it later" deferral.

## What "doing the job well" looks like

A finished setup is:

1. Identity established — auth works, theme preference picked (Light / Dark / System), display name + avatar.
2. Workspace either created (lab admin path) or joined (invited member path).
3. Optional: OSF account connected for later preregistration push.
4. Optional: provider connections (Prolific, CloudResearch) deferred until first recruitment need.
5. The user lands on Studies destination ready to act, not on yet another setup screen.

## Current alternatives the user hires

- **Qualtrics signup + lab Doc onboarding** — minimal account, all real onboarding via Slack/Doc handed by a lab mate.
- **PsychoPy + GitHub** — no account onboarding per se; setup is git clone + dependencies.
- **OSF account first, tooling later** — common pattern; this product needs to slot in alongside.

# ADR 0069 — Audio-stimulus block + Octave TTS generation & caching

- **Status:** accepted
- **Date:** 2026-06-25
- **Deciders:** Paweł Rosner (project owner)
- **Tags:** ai, blocks, tts, storage, cost-model, vendor-seam

## Context

V2.1 H5 adds a **stimulus** block (`core/audio-stimulus@1.0.0`): the researcher writes a script + a delivery direction, and Hume **Octave TTS** generates the audio the participant hears. It's a delivery block (no participant response), enabling voice-condition × content factorial designs (e.g. the same headline read in a trustworthy vs. an anxious voice). It builds on substrate already shipped: the AI gateway (ADR-0066) with `runTts()`, the verified `humeAdapter.synthesizeAudio` (ADR-0067, `POST /v0/tts` → base64), the module/registry + Builder/Take rendering (ADR-0012), and R2 storage (ADR-0003).

Two things make this a new pattern worth an ADR rather than "just another block": (1) audio is **generated at author time, not authored** — a server action calls a paid vendor and persists bytes to R2; and (2) generation is **expensive enough to cache**, so identical inputs must not re-bill. ADR-0061's `ai-chat` runs per-participant at run time; audio-stimulus runs per-(script,voice) at publish/preview time, a different cost + lifecycle shape. We pin the generation flow, the cache key, and the storage location here so H5's code and any future TTS-generating block share one contract.

## Options considered

### Option A — Generate server-side via `runTts`, cache in R2 by a content hash, store the key in block config (chosen)

- A `studies.generateStimulusAudio` mutation hashes `(script, description, voicePresetId)` → a deterministic R2 key under `ws/<workspace>/audio-stimulus/<hash>.mp3`. If the object already exists (HEAD), it's a **cache hit** (no vendor call, no cost); otherwise it calls the gateway `runTts()` (which audits + meters + enforces the budget cap), PUTs the returned base64 to R2 via a presigned upload, and returns the `/api/media/<key>` URL. The Builder stores that URL on the block config; the Take surface just plays it (no participant key needed at run time).
- **Pros.** Reuses every existing seam (gateway audit/cost, storage presign, registry/Take); caching makes iterative authoring free after the first generation; generated audio lives in `ws/` (researcher content, public-served like other stimuli) — not `resp/` (no participant PII); cost is bounded (per-(script,voice), not per-participant).
- **Cons.** A new server action that both calls a vendor and writes storage; needs a content-hash convention; participants depend on the generated object existing at run time (publish-time generation must succeed).

### Option B — Generate on the participant's device / at run time, per participant

- Call TTS when each participant reaches the block.
- **Pros.** No author-time step.
- **Cons.** Pays per participant for identical audio (Nx cost); adds latency + a failure mode mid-study; needs the workspace key reachable from the run path. Rejected — wrong cost + reliability shape for a fixed stimulus.

### Option C — Researcher uploads pre-recorded audio (no TTS)

- Reuse the existing media uploader; no Hume.
- **Pros.** Simplest; no vendor.
- **Cons.** Doesn't deliver the value prop (programmatic emotional delivery / per-condition voice variants from one script). That's the existing audio upload, not this feature. Rejected (it already exists).

## Decision

**We will ship `core/audio-stimulus@1.0.0` as a delivery block whose audio is generated by a `studies.generateStimulusAudio` server action (Option A): hash `(script, description, voicePresetId)` → a deterministic R2 key under `ws/<workspace>/audio-stimulus/`; on a cache miss call the AI gateway `runTts()` (audited, metered, budget-enforced) and PUT the bytes to R2; store the resulting `/api/media` URL on the block config so the Take surface plays it with no run-time vendor call.**

Plainly: the researcher types a line and a delivery note and clicks Generate; we make the clip once, file it in R2 under a fingerprint of its inputs, and remember the link. Click Generate again with the same inputs → instant, free (cache hit). Participants just hear the saved clip — no key, no per-person cost, no live dependency on Hume during the run.

## Consequences

- **What becomes easier.** Voice/emotion stimuli without a recording studio; per-condition variants from one script; iterative authoring is free after first generation; cost is visible in the same `ai_invocation` ledger as every other AI call.
- **What becomes harder.** A new author-time async step (generation can fail / take seconds) needs clear Builder status UI; the cache key must include every input that changes the audio or a stale clip could be reused.
- **What we are now committed to.** Generation goes through `runTts()` (never the adapter directly); generated audio lives in `ws/` keyed by content hash; the block stores a media URL, not raw audio; the voice-preset catalog (10 vetted presets) is the only voice selector exposed (full Octave catalog = V2.2).
- **What we are now precluded from.** Per-participant TTS for a fixed stimulus; storing generated audio in `resp/` (that path is participant data); exposing arbitrary Hume voice IDs in the Builder.

## Amendment (2026-06-26) — per-variant audio generation

A factorial variant can bind the audio-stimulus `script` to a factor (different script per level). Previously the audio (`audioUrl`) was generated once from the base script, so every variant played the same clip — an offered-but-broken customization. Fix (ADR-0058 territory): a new `studies.generateAudioClip({studyId, script, description?})` mutation renders an arbitrary script to a cached `/api/media` clip and returns the URL **without mutating any block** (same R2 cache key as the single-block generator, so identical scripts are free). The variants editor's "Generate audio for each level" button calls it once per level's script and stores the URLs as a system-managed **`audioUrl` variant binding**; the runtime's existing per-cell `resolveConfigForCell` then plays each variant its own clip. `audioUrl`/`audioHash` are also marked `derivedFields` so they're hidden from the manual varying-field picker (you vary the script; the audio follows). No schema/migration change (variant bindings live in `definitionSnapshot`).

## Revisit triggers

- Per-participant dynamic TTS (e.g. personalized name in the audio) becomes a requirement — that breaks the author-time cache model.
- Octave pricing moves to a shape the advisory `ttsCostUsdFromChars` can't approximate.
- We need streaming TTS (long scripts) rather than one synchronous clip.
- The voice-preset catalog graduates to the full Hume voice library (V2.2).
